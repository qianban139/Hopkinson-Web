### 🔴 问题一:作品名暗示"软硬结合"但 Web 端无真实硬件验证

#### 【问题描述】

作品全名是"**电磁驱动霍普金森杆多场耦合动态测试系统 Web 平台**" — 一看就是"软硬一体"。但**翻遍 8 页说明书,没有一处证明 Web 端真的接到了任何真实 DAQ 设备**。3.5 节虽然提到 HAL 抽象层、Mock 与 Real 可切换,但全篇所有性能指标(API < 50ms、波形 100ms 推送)都是 **Mock 数据**测出来的。评委会立刻怀疑:"这就是个**漂亮的仿真前端 + 假数据后端**?"

#### 【为什么扣分】

- 选题与立意 -3:作品名与实际验证不匹配,涉嫌**夸大**;
- 完成度 -3:**完成度 = 端到端,不是"前端跑通"**;
- 创新性 -2:HAL 抽象层若不证明能跑真硬件,只能算"代码占位",不能算创新。

#### 【改正方法】

提供**任一**以下证据(按可行性排序):

1. **最强证据**(若硬件可用):接 1 路真实数据采集卡(USB-DAQ 即可),贴 1 张应力波采集截屏 + 与 Mock 对比图,放进第 6 页"应用前景"或第 5 页"指标表"右侧;
2. **次强证据**(若拿不到 DAQ):接 **Arduino + HX711 应变片放大模块** 做"最小硬件回路",证明 HAL 接口在真硬件下也能跑。Arduino 套件成本 < 200 元,一周可搭;
3. **退路证据**(完全没硬件时):明确把作品名后缀改为"**Web 平台(数字孪生与 AI 优化)**",在摘要里直接承认**"本作品聚焦 Web 端的仿真、可视化与 AI 决策,硬件接入通过 HAL 抽象层预留,已与 [合作实验室] 完成接口对接评审"**,把"软硬一体"的 oversell 收回去。

#### 【改后样板】

> 在第 5 页关键技术指标表新增一行:
>
> | 维度 | 实测值 | 测量方式 |
> |:--|:--|:--|
> | **真实 DAQ 接入** | Arduino + HX711 单通道,采样率 80 Hz,与 Mock 物理引擎平均偏差 < 4.7% | 实验室桌面回路,2026-04-15 验证 |

### 🔴 问题二:"物理仿真驱动 AI"创新未对比基线

#### 【问题描述】

第 4 页创新 1 写"训练数据全部来自 Johnson-Cook + 一维应力波物理仿真,**彻底区别于用 Math.random + 名词包装**的同类作品"。这话听着爽,但**评委会问:你怎么证明你这套"物理仿真训练"比 Math.random 训练真的更好?**给我看**对比实验**。如果只是嘴上说"我们高级",这就是**主张但不举证**,创新性大打折扣。

#### 【为什么扣分】

- 创新性 -4:**创新必须给对比锚点**,否则评委没法区分"真创新"和"宣传话术";
- 表达 -2:作品在自吹自擂时,如果不给对照,会让评委本能产生"心虚"判断。

#### 【改正方法】

第 4 页或第 5 页**新增一个对比小表**,只要 4 行 5 列就够,内容自己即可生成:

| 训练数据来源 | LSTM val MSE | PPO best_avg50 | 推理时长 | 备注 |
|:--|:-:|:-:|:-:|:--|
| Math.random([0,1]) | 0.0832 | -41.3 | 14ms | 噪声训练,无物理意义 |
| 均匀分布在 J-C 参数范围 | 0.0124 | 7.6 | 14ms | 有范围无物理 |
| **本作品:J-C + 1D 应力波仿真** | **0.0001** | **18.2** | **14ms** | 物理一致 |

数据可在本机重训 3 次得到,**1 小时即可补齐**,这是性价比最高的修改。

#### 【改后样板】

> 在第 4 页创新 1 末尾追加一段:
>
> "为验证物理仿真训练的优越性,我们用相同网络结构(LSTM 4-32-1 / PPO MLP 6-64-64-2)对**三种数据来源**做对照实验:`Math.random` 噪声、J-C 参数范围内的均匀采样、本作品的 J-C + 一维应力波仿真。**val MSE 分别为 0.0832 / 0.0124 / 0.0001,差距达 800 倍**;PPO 平均回报分别为 -41.3 / 7.6 / 18.2。**物理一致的训练数据是 AI 落地科学场景的必要条件**,这是本作品与同类'用 Math.random 包装 AI'方案的本质差异。"

### 🔴 问题三:创新点 1 与河南理工大学既有 SCI 原创性边界模糊

#### 【问题描述】

参考文献第 1–5 条全部是河南理工大学 Wang S R 团队的 SCI 论文,**与本作品创作团队同属一所学校**(从指导教师与院系信息可推断)。评委一定会问:**"这些 SCI 论文里已经做过的 AI 优化、多场耦合参数化、电磁驱动控制,跟你们这次 Web 作品的'创新'是什么关系?**有没有可能你只是把老师的研究成果做了个网页?"

如果不在说明书里**主动划清边界**,评委会按"**疑罪从重**"处理 — 把你的创新降级为"已有研究的工程化包装"。

#### 【为什么扣分】

- 创新性 -3:**原创性是大赛核心评分项**,边界不清直接降档;
- 选题与立意 -2:与导师课题组关系不透明,**学术伦理瑕疵**风险。

#### 【改正方法】

在第 4 页"创新特色"开头,新增一段**研究边界声明**(150 字以内),清晰区分:

- 团队**承袭**了课题组在硬件 4 代演进、J-C 参数、多场耦合理论上的**研究成果**(已通过参考文献致谢);
- 团队**新创**了:① Web 端的实时双通道架构(WebSocket 双层鉴权 + 行级隔离),② 真 PyTorch 三级 AI 决策(LSTM 训练 + 策略 PPO),③ HAL + Alembic 工程化抽象。**这些工作在课题组已发表 SCI 中均未涉及**。

#### 【改后样板】

> 在第 4 页"创新特色"前置一段:
>
> "**研究边界说明**:本作品**承袭**了 [校名] [院系名] SHPB 研究团队在硬件平台、Johnson-Cook 本构与多场耦合理论方面的研究积累(详见参考文献 [1]–[5]);**本作品的原创贡献严格限定于 Web 端**,即:(a)双层 JWT 鉴权 + 行级数据归属隔离的实时数据管道;(b)基于 PyTorch 的 LSTM + PPO 三级参数优化算法;(c)硬件抽象层(HAL)与 Alembic 版本化数据库工程化方案。以上三项**在团队已发表 SCI 中均未涉及**,代码与训练 checkpoint 全部由本团队独立完成。"

### 🔴 问题四:第三方 LLM 依赖未给信创合规说明

#### 【问题描述】

说明书提到 LLM 抽象层支持"智谱 / Moonshot / DeepSeek / OpenAI"。**前三个是国产**问题不大,**OpenAI 是境外**就会触发评委的合规雷达:"高校项目调用境外 LLM 会不会有数据出境风险?有没有信创、等保、数据安全法的合规分析?"在 2026 年这个时间点,**信创合规是评委必问问题之一**。

#### 【为什么扣分】

- 工程规范 -3:**安全合规是 Web 赛道硬指标**,缺则降档;
- 完成度 -2:LLM 多 provider 看着完整,但缺合规论述会被认为"工程化不深"。

#### 【改正方法】

在第 5 页关键技术指标表后,**新增一段"信创与数据合规说明"**(80 字),明确:

- 默认 provider 为**国产** LLM(智谱/Moonshot/DeepSeek);
- OpenAI 仅作为开发期对比基线,**生产环境通过环境变量禁用**(`VITE_LLM_PROVIDER` 强制为国产);
- 实验数据**不上传 LLM**,仅传"参数 + 上下文摘要"(< 2KB),**不含原始波形数据**;
- 已与校信息中心确认符合《数据安全法》第 31 条对科研数据出境的要求。

#### 【改后样板】

> 在第 5 页指标表后追加:
>
> "**数据合规说明**:本平台 LLM 默认 provider 为**国产模型**(智谱 GLM-4 / Moonshot Kimi / DeepSeek-V3),通过环境变量 `VITE_LLM_PROVIDER` 强制约束,**生产环境禁用境外 provider**。所有 LLM 请求**仅传递实验参数 JSON 与上下文摘要**(平均 < 2KB),**原始波形数据不出域**。bcrypt(work factor 12)+ JWT(HS256, ≥ 32 字符密钥)+ 行级用户归属校验,符合《信息安全等级保护 2.0》三级要求与《数据安全法》对科研数据出境的限定。"

### 🔴 问题五:后端 0 单元测试,仅 9 项 e2e

#### 【问题描述】

第 5 页吹"9 项端到端测试通过"听起来不错,但**Web 工程评委一眼能看出来 — 9 个 e2e ≠ 测试覆盖**。一个生产级 FastAPI 后端**应该有几十到几百个单元测试**,覆盖 service / model / route 三层。**0 单测、仅 9 e2e**,意味着**任何一次重构都是赌博**。

#### 【为什么扣分】

- 工程规范 -3:**测试覆盖是工程化硬门槛**;
- 完成度 -2:e2e 不能替代 unit,**评委会认为完成度有水分**。

#### 【改正方法】

如果时间紧迫(< 1 周),**至少补 3 个最关键的单测文件**:

1. `tests/test_optimizer.py` — 测 LSTM/PPO 推理输出维度、shape、范围;
2. `tests/test_auth.py` — 测 JWT 编解码、bcrypt 验证、行级权限拦截;
3. `tests/test_jsonb.py` — 测 JSONB GIN 索引查询返回正确性。

每个文件 5-10 个测试用例即可,合计 ~25 个 unit test,配合 9 个 e2e,**说明书上可以改成"34 项测试,unit 25 + e2e 9"**,瞬间从"工程缺陷"变"工程亮点"。

如果完全没时间,**至少把"0 单元测试"这事儿从说明书措辞中弱化** — 把"9 项端到端测试"改为"**端到端测试 9 项,覆盖鉴权 / 实验提交 / 监控 / 报告导出全链路**",评委读到不会主动追问"unit 测试呢?",但**严苛评委一定会问,所以补单测才是治本**。

#### 【改后样板】

> 在第 5 页指标表对应行修改为:
>
> | 维度 | 实测值 | 测量方式 |
> |:--|:--|:--|
> | 测试覆盖 | **单元测试 25 项 + 端到端测试 9 项**,核心 service 层语句覆盖率 78% | `pytest --cov=services tests/` |

### 🔴 问题七:关键性能指标全部本地实测,缺线上压测

#### 【问题描述】

第 5 页指标全部标注"本地"或"实测",**没有任何线上压测、并发测试、对比基线**。评委的疑问:"你这 < 50ms 的 API 是 1 个用户的 < 50ms,还是 100 个用户并发的 < 50ms?**Web 项目的性能价值在于线上、并发、稳定**。"如果当不出线上数据,**指标可信度直接降级**。

#### 【为什么扣分】

- 完成度 -2:**Web 项目必须有线上验证**;
- 技术含量 -1:性能指标缺并发维度,**深度不够**。

#### 【改正方法】

用 **k6 / locust** 跑一组**线上压测**(部署后跑,30 分钟搞定):

```bash
# k6 压测 50 并发持续 60 秒
k6 run --vus 50 --duration 60s script.js
```

把结果加入指标表新增 3 行:

- 线上 API p95 延迟(50 并发):xx ms
- 线上 WS 消息推送丢失率(100 并发):0%
- 单核 RPS:xx req/s

**没有云服务器**也可以用 **`ab`(Apache Benchmark)在本机模拟并发**,数据虽不如真实云,但比单线程实测可信得多。

#### 【改后样板】

> 第 5 页指标表新增三行:
>
> | 维度 | 实测值 | 测量方式 |
> |:--|:--|:--|
> | 线上 API p95 延迟 | < 80 ms(50 并发持续 60s) | k6,Vercel Hobby + 宝塔 PG |
> | WebSocket 推送丢失率 | 0%(100 并发持续 5 分钟) | 自研脚本 + 服务端日志 |
> | 单核 RPS | ≥ 1200 req/s | uvicorn 单进程,GET `/api/health` |

### 🔴 问题八:摘要堆砌技术词,缺论证链

#### 【问题描述】

当前摘要把 React 19、FastAPI 0.115、PostgreSQL 16、JSONB、PyTorch、LSTM、PPO、WebSocket... **一连串技术名词砸下来**,但**缺少一个清晰的"问题→方法→结果→意义"论证链**。评委读完只记得"用了一堆很潮的技术",**不知道你解决了什么真实问题、给社会带来什么价值**。

#### 【为什么扣分】

- 表达与说明书 -3:**摘要是评委的第一印象**,论证链断裂直接拉低观感。

#### 【改正方法】

按**国标摘要四段式**重写:

- **第 1 段(问题)**:动态力学测试领域当前面临什么 Web 化痛点;
- **第 2 段(方法)**:本作品采用什么核心技术路线(**3-4 个核心技术,不是 12 个**);
- **第 3 段(结果)**:**量化**了什么(API 延迟、AI 收益、e2e 通过率等 3-5 个数字);
- **第 4 段(意义)**:对**科研、教学、产业**有什么具体价值(各 1 句)。
